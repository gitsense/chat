# MetadataInsightResult Architecture

## Overview

The MetadataInsightResult component is a standalone component for visualizing and interacting with metadata insights analysis results. It can render in any container (message body or modal) and provides filtering, search, and pagination capabilities.

## Data Contract

### Input Format

The component receives raw markdown in the "pretty format" generated by the metadata search service:

```markdown
# Metadata Insights Analysis

## Repository Reference
- **R0**: Tutorial/hello-world - Repository: Tutorial/hello-world
- **R1**: Tutorial/hey-world - Repository: Tutorial/hey-world

## Branch Reference
- **B0**: main

## Analyzer Reference
- **A0**: tiny-overview::file-content::default - Tiny Overview

## Field Reference
- **F0**: purpose (string) - Extracted by A0. A single sentence describing the file's primary purpose, derived from the first sentence of the Summary.

## Data (File Metadata)
| Repo | Branch | File | Language | Chat ID | F0 |
| ------ | -------- | ------ | --------- | ------- | --- |
| R0 | B0 | hello-world.php | php | 9 | "This is a simple..." |
| ... | ... | ... | ... | ... | ... |

```

## Architecture Principles

### 1. Immutability

- **Parsed data is immutable**: Once the parser creates the data structure, it never changes
- **UI components work with copies**: Components receive data and create filtered copies as needed
- **No direct mutation**: State changes create new objects rather than modifying existing ones

### 2. State Separation

#### Persistent State (saved to tool block)
- Filter selections (repositories, branches, analyzers, fields)
- These are preserved across sessions and page reloads
- Stored in the `metadata-insights-result` tool block configuration

#### Transient State (reset on load)
- Current page number
- Search query text
- Items per page
- These are reset to defaults when the component loads

### 3. Container Agnostic

- The core viewer component can render in any DOM container
- Same component works in message body or full-screen modal
- Container switching is handled at the manager level, not component level

### 4. Component Responsibilities

- **MetadataDataParser**: Transforms markdown to structured data
- **MetadataResultStateManager**: Manages filter state and data filtering
- **MetadataResultViewer**: Core container-agnostic viewer
- **MetadataResultPreview**: Minimal preview for message view
- **MetadataFilterPanel**: Filter controls with aggregated counts
- **MetadataDataTable**: Searchable, paginated table
- **MetadataActionPanel**: Action buttons with confirmations
- **MetadataMessageRewriter**: Rewrites messages with filtered data

### 5. Error Handling

- Graceful degradation for malformed markdown
- Empty state handling for no data
- Validation of required fields
- User-friendly error messages

## Data Flow

```
Raw Markdown → Parser → Immutable Data → State Manager → Components
```

1. Parser transforms markdown to immutable data structure
2. State manager receives data and initial filter state
3. State manager provides filtered data to components
4. Components emit events for filter changes
5. State manager updates filters and provides new filtered data
6. Apply filter action triggers message rewriter
7. Message rewriter creates new markdown with filtered data
```

## Performance Considerations

- Lazy rendering of large datasets
- Debounced search input
- Efficient filtering algorithms
- Minimal re-renders through immutable data
