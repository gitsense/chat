## 1. Create a "Brain"

The foundation of smarter agents is a specialized analyzer, a "Brain" that understands your code at a deeper level than AI working without domain context. Instead of treating all code the same, a Brain extracts structured metadata that reveals intent, purpose, and relationships.

**See Examples:**
- [Product Manager] - Encode business logic so agents understand revenue impact
- [QA Engineer] - Define risk levels so agents prioritize critical paths
- [DevOps Engineer] - Map infrastructure dependencies so agents avoid cascading failures

**Why This Matters**

Analyzers make it possible to map your codebase in ways that are currently impossible. You don't need to be a programmer to encode domain expertise to make agents smarter, you just need to know your domain.

**How Analyzers Gain Context**

You might wonder how an analyzer knows that a specific file is "revenue-critical" or part of the "checkout flow" just by looking at the code. Often, this critical context lives outside the file itself-in documentation, architecture diagrams, or team wikis.

When creating a Brain, you can specify **Reference Files**. These are documents (like a business logic map or a feature specification) that the analyzer loads alongside every file it reviews. This provides the analyzer with the "big picture" context it needs to accurately tag files based on your domain knowledge, rather than just guessing from the code syntax.

**Combine Brains for Deeper Insights**

A single Brain is powerful, but multiple Brains working together are transformative. Just as you might ask both a security expert and a compliance officer to review the same code, you can orchestrate multiple analyzers to examine your data from different angles.

### Examples

#### Product Manager

**The Scenario:**
A Product Manager notices that agents keep suggesting changes to files without understanding the business impact. They need agents to know which features are "revenue-critical" vs. "nice-to-have."

**The Solution (No Coding Required):**
The Product Manager creates a Brain by defining business rules:

> "When you analyze a file, tell me:
> 1. Which business feature does this support? (e.g., Checkout, User Profile, Settings)
> 2. Is this revenue-critical? (Yes/No)
> 3. What happens if this breaks? (Users can't pay, Users can't login, etc.)"

**The Result (Enriched Data):**
```json
{
  "business_feature": "Checkout",
  "revenue_critical": true,
  "impact_if_broken": "Users cannot complete purchases",
  "file": "backend/src/payment/processPayment.js"
}
```

**The Agent Benefit:**
Now agents can query: *"Show me all revenue-critical files in the Checkout feature"* to prioritize their work, something impossible with grep.

#### QA Engineer

**The Scenario:**
A QA engineer notices that bugs in the "checkout flow" keep getting missed because agents can't distinguish between:
- Files that handle *payment processing* (critical)
- Files that handle *UI styling* (not critical for payment logic)
- Files that handle *error messages* (important but different from payment logic)

**The Solution (No Coding Required):**
The QA engineer opens GitSense Chat and creates a Brain by simply describing what they see:

> "When you analyze a file, tell me:
> 1. Is this part of the payment processing flow? (Yes/No)
> 2. What's the risk level if this breaks? (Critical/High/Medium/Low)
> 3. What external services does it depend on? (Stripe, PayPal, etc.)"

**The Result (Enriched Data):**
```json
{
  "payment_flow": true,
  "risk_level": "Critical",
  "external_dependencies": ["Stripe API"],
  "file": "backend/src/payment/processPayment.js"
}
```

**The Agent Benefit:**
Now agents can query: *"Find all Critical payment files that depend on Stripe"* instead of blindly grepping for "payment" and getting 200 irrelevant results.

#### DevOps Engineer

**The Scenario:**
A DevOps engineer is tired of agents suggesting changes to infrastructure files without understanding the downstream impact. They need agents to know which services depend on which databases or external APIs to prevent cascading failures.

**The Solution (No Coding Required):**
The DevOps engineer creates a Brain focused on dependencies:

> "When you analyze a file, tell me:
> 1. Does this file define a service endpoint? (Yes/No)
> 2. What databases does it connect to? (Postgres, Redis, etc.)
> 3. What external APIs does it call? (Stripe, SendGrid, etc.)"

**The Result (Enriched Data):**
```json
{
  "service_endpoint": true,
  "databases": ["Postgres"],
  "external_apis": ["Stripe"],
  "file": "backend/src/payment/processPayment.js"
}
```

**The Agent Benefit:**
Agents can now query: *"Find all services that connect to the Postgres database"* to assess the blast radius of a database migration, ensuring they don't break critical infrastructure.

## 2. Load Your Data

Now that you have a Brain, you need data to feed it. GitSense Chat treats Git as a first-class citizen, making it trivial to bring your code into a structured, queryable format. Whether you have a dozen loose files or thousands spread across repositories, your entire codebase becomes easily accessible and referenceable.

**Why This Matters**

Your agents need access to your codebase, but not all at once. Different workflows require different import methods. For quick experiments or ad-hoc analysis, you need something fast and frictionless. For complete repositories where file hierarchy matters, you need a tool that preserves structure. GitSense Chat provides both.

### Three Ways to Import Your Code

GitSense Chat offers three distinct import methods, each optimized for different scenarios:

#### **Dropzone: Quick File Uploads**

Best for rapid experimentation and small-scale imports. The Dropzone is a browser-based interface that lets you drag-and-drop files or paste text snippets directly into your chat.

#### **Manual (CLI): Power User Imports**

Best for importing existing Git repositories or complete directory structures from the command line. This method gives you full control and creates permanent repositories.

#### **Direct (Enterprise): Large-Scale Imports**

Best for enterprise environments needing to import massive codebases asynchronously. This method runs imports as background jobs and supports GitHub URLs directly. Requires Enterprise component installation.

### Quick Exercise: Try the Dropzone

Let's experience how effortless it is to import data using the Dropzone method:

1. Locate the **Import Data** button in the panel on the right side of your screen.
2. Click it and select **New Message** from the tooltip menu (this keeps your lesson chat open for reference).
3. In the new message, ensure the **Dropzone** tab is selected (it's the default).
4. Notice the toggle at the top: you can either drag files or create a text snippet. For this exercise, click **Create Text Snippet**.
5. Enter a filename with extension: `hello-agent.js`
6. Paste this content into the text area:

  console.log(`Hello, ${name}! Ready to analyze some code?`);

7. The **Repository Configuration** section appears automatically. Enter a name prefix like `agent-demo`.
8. Review the file list-you'll see your `hello-agent.js` ready for import.
9. Click **Import Files** and confirm in the dialog.
10. Watch the progress bar as your files upload and import.
11. Once complete, check the **Repositories** tree in the right panel-expand it to find `agent-demo-[random-id]` and your file inside.

Your code is now a native, queryable entity that agents can reference by path and chat ID.

## 3. Analyze at Scale

With Git as a first class citizen, importing repositories with thousands of files is trivial. This section shows you how to process them. Analyzing a few files is simple. Analyzing thousands of files across multiple repositories is where the real challenge lies. This is where GitSense Chat's batch processing capabilities become essential.

**Scale Without Specialists**

Consider this scenario: An accountant has thousands of client documents in text format. They have built an analyzer that extracts financial categories and risk flags. Now they need to process all documents. They do not need an ML team or complex infrastructure. They need a way to run their analyzer at scale and monitor the progress.

**The Analyze Batch Builder**

The Analyze Batch Builder is your command center for deploying analyzers at scale. It handles the complexity of processing large volumes of files while giving you clear visibility into the process.

You configure your analysis in five sections: Analyzer, Reference Files, Files to Analyze, File Filters, and Batch Configuration. After configuration, you launch batches and track progress in real time. Every analysis creates an "Analyze Chat" that allows you to review the system prompt, instructions, processed files, and extracted metadata. You can also use the Fork tab to create variations of your analysis with different models or analyzers for comparison.

### Why This Matters

The Batch Builder turns analysis into a conversation. Every batch runs as a chat, letting you question the AI about its decisions and refine your analyzer instructions based on actual results. This iterative loop helps you build better analyzers without guesswork.

GitSense Chat democratizes data analysis. No technical skills are required. Since everything is a chat, you can just ask AI.  

This means domain experts-accountants, lawyers, product managers-can analyze thousands of documents without writing code. They simply chat with the AI for help, launch batches, and review results through an interface that explains itself.

### Quick Exercise

1. Click the **Analyze** button in the right panel then press 'Go' when prompted. This will replace your current web page. Press your browser's back button to come back to this document..

2. Configure your analysis in five sections:
   - **Analyzer**: Select the analyzer that will process your files. This is required first.
   - **Reference Files**: Optional files that provide context for every batch (READMEs, docs, etc.).
   - **Files to Analyze**: Select the specific files you want to process.
   - **File Filters**: Refine your selection by commit time, language, size, or analysis status. Since everything is stored in Git, you can filter by commit date to enable incremental analysis.
   - **Batch Configuration**: Define how files are grouped into batches (max files, tokens, size).

3. Switch to the **Batches** tab to see your analysis batches. Click any batch to launch it in a new tab. Watch progress update in real time. If a batch fails, click retry.

**GitSense Chat solves this by letting you transfer your domain knowledge at scale.**

**Now, in Step 4, you reap the reward.** That metadata becomes the intelligence layer that agents query. Instead of guessing, they *know*, because *you* taught them.
